# Parksy APK Lab — 소스풀 & SCM 프로세스 백서

> **작성일:** 2026-03-01
> **문서 성격:** 소스 확보 → 편집 → 빌드 → 콘텐츠화에 이르는 전체 파이프라인의 프로세스 정의서 겸 법적 방어 논리서
> **한 줄 요약:** F-Droid 오픈소스 풀 + 공식 SDK + AI 편집 = Parksy 앱 제조 공정

---

## 0. 이 문서의 목적

Parksy 앱 시리즈 개발에 있어 소스코드를 어디서 가져오고, 어떻게 관리하고, 어떤 법적 근거로 사용하고, 최종적으로 어떤 형태(앱 / YouTube 콘텐츠)로 아웃풋하는지를 정의한다.

이 문서 하나로 다음 질문에 전부 답할 수 있어야 한다:

1. 코드를 어디서 가져오는가?
2. 그것이 합법인가?
3. 어떤 프로세스로 편집하는가?
4. 결과물은 어떻게 배포되는가?
5. YouTube 콘텐츠화 시 법적 리스크는 없는가?

## 1. 소스풀 아키텍처 (3-Tier)

### Tier 1: F-Droid 오픈소스 풀 (Primary)

```
역할:    베이스 코드 확보 + 구현 참조
법적 근거: 각 앱의 오픈소스 라이선스 (Apache 2.0, MIT, GPL 등)
방법:    git clone → 분석 → fork/참조 → 편집
```

**F-Droid란:** 오픈소스 Android 앱 전용 앱스토어. 등록 조건이 "소스코드 100% 공개"이며, F-Droid 서버가 직접 소스에서 빌드하여 APK를 배포한다. 4,000개 이상의 검증된 오픈소스 Android 앱이 있다.

| 비교 항목 | GitHub | F-Droid |
|-----------|--------|---------|
| 대상 | 모든 종류의 코드 | Android 앱 전용 |
| 오픈소스 필수 | 아님 (private repo 가능) | **100% 필수** |
| 빌드 검증 | 없음 | **서버에서 직접 소스 빌드** |
| 결과물 | 코드 | 코드 + 설치 가능 APK |
| 품질 보증 | 없음 | 소스 = 바이너리 일치 검증 |

**핵심 가치:** 디컴파일 필요 없이 원본 소스코드, 주석, 커밋 히스토리, 이슈 토론까지 전부 확보 가능하다.

#### Parksy 앱별 F-Droid 소스맵

| Parksy 앱 | F-Droid 참조 앱 | 라이선스 | 참조 포인트 |
|-----------|----------------|----------|------------|
| Wavesy (음원 편집) | Ringdroid | Apache 2.0 | 파형 편집, 오디오 자르기 |
| ChronoCall (STT) | whisperIME, Transcribro | Apache 2.0 | STT 파이프라인, 음성 인식 |
| TTS Factory | sherpa-onnx (k2-fsa) | Apache 2.0 | TTS + STT + 화자분리 통합 |
| Pen (오버레이) | DrawAnywhere | — | 오버레이 권한, 캔버스 드로잉 |
| Capture (클립보드) | Clipboard Cleaner | — | 클립보드 감시 로직 |

#### sherpa-onnx 특별 주목

- **라이선스:** Apache 2.0 (상업 사용 포함 자유)
- **GitHub Stars:** 10,400+
- **커버리지:** STT + TTS + Speaker Diarization + VAD
- **의미:** ChronoCall의 유료 Whisper API, TTS Factory의 Google Cloud TTS, 향후 화자분리 기능을 모두 이 하나의 라이브러리로 대체 가능
- **온디바이스 추론:** 서버 없이 폰에서 직접 실행 → API 비용 제로

### Tier 2: 삼성 공식 SDK & API (Secondary)

```
역할:    기기 최적화 + 플랫폼 특화 기능
법적 근거: 삼성 개발자 공식 문서 및 SDK 라이선스
방법:    공식 문서 참조 → API 호출 → 기기별 테스트
```

| 참조 대상 | 용도 | 출처 |
|-----------|------|------|
| One UI API | 삼성 고유 UI 패턴 | developer.samsung.com |
| S Pen SDK | 필압, 제스처, 오버레이 | Samsung S Pen Framework |
| 녹음 폴더 경로 패턴 | 통화 녹음 파일 탐지 | 공식 문서 + 커뮤니티 검증 |
| Good Lock / Sound Assistant | UI/UX 패턴 참조 | 삼성 공식 앱 |

### Tier 3: 상용 앱 구조 참조 (Tertiary — 제한적)

```
역할:    UI/UX 패턴 및 아키텍처 참조 (코드 복사 아님)
법적 근거: 리버스 엔지니어링의 제한적 합법성 (개인 학습 목적)
방법:    ADB + jadx → 구조 분석 → 개념만 참고 → 독자 구현
```

**이 Tier는 보조적 수단이다.** Tier 1 (F-Droid)에서 해결 가능하면 Tier 3에 가지 않는다.

## 3. 법적 방어 논리

### 3.1 Tier 1 (F-Droid 오픈소스) — 완전 합법

```
근거: 각 프로젝트의 오픈소스 라이선스
결론: 사용, 수정, 배포, YouTube 공개 모두 합법
조건: 라이선스 고지 의무 준수
```

| 라이선스 | 코드 사용 | 수정 배포 | YouTube 공개 | 조건 |
|----------|----------|----------|-------------|------|
| Apache 2.0 | O | O | O | NOTICE 파일 포함 |
| MIT | O | O | O | 저작권 고지 포함 |
| GPL v3 | O | O* | O | *배포 시 소스 공개 의무 |
| BSD | O | O | O | 저작권 고지 포함 |

**GPL 방어 전략:**
- GPL 코드를 직접 복사하여 Parksy에 포함 → Parksy도 GPL 공개 의무 발생
- GPL 코드를 **참조만** 하고 직접 새로 작성 (클린룸 구현) → 의무 없음
- **원칙: Apache 2.0 / MIT 우선 채택, GPL은 참조만**

### 3.2 Tier 2 (삼성 공식) — 합법

```
근거: 공식 SDK 라이선스, 개발자 문서 이용약관
결론: SDK 사용 및 API 호출은 의도된 용도
조건: SDK 라이선스 조건 준수
```

삼성이 공식으로 제공하는 SDK와 문서를 참조하는 것은 해당 SDK의 존재 이유 그 자체다.

### 3.3 Tier 3 (상용 앱 디컴파일) — 조건부 합법

```
근거: 저작권법 상 리버스 엔지니어링의 제한적 허용
결론: 개인 학습 목적으로 구조 분석은 허용
조건: 코드 복사/재배포 금지, 보호 기술 우회 금지
```

#### 한국 저작권법 근거

**저작권법 제101조의4 (프로그램코드역분석):**
> 정당한 권한에 의하여 프로그램을 이용하는 자 또는 그의 허락을 받은 자는
> 호환에 필요한 정보를 쉽게 얻을 수 없고 그 획득이 불가피한 경우에
> 해당 프로그램의 호환에 필요한 부분에 한하여 프로그램코드역분석을 할 수 있다.

**적용:**
- "정당한 권한에 의하여 프로그램을 이용하는 자" → 합법적으로 설치한 앱의 사용자
- "호환에 필요한 정보" → 삼성 녹음 폴더 경로, Intent 규격 등
- Parksy ChronoCall이 삼성 녹음 앱과 연동하기 위한 분석 → 호환성 목적

#### 방어 라인

| 행위 | 합법 여부 | 근거 |
|------|----------|------|
| 내가 설치한 앱의 APK 추출 | O | 정당한 사용자의 백업 권리 |
| AndroidManifest.xml 구조 분석 | O | 호환성 정보 확인 |
| UI/UX 패턴 관찰 후 독자 구현 | O | 아이디어는 저작권 보호 대상이 아님 |
| 디컴파일 코드를 통째로 복사 | **X** | 저작물 복제 |
| 디컴파일 코드를 YouTube에 전체 공개 | **X** | 저작물 공중 송신 |
| 보호 기술(DRM 등) 우회 | **X** | 기술적 보호 조치 무력화 금지 |

**원칙: Tier 3는 "개념 이해"용이지 "코드 복사"용이 아니다.**

### 3.4 YouTube 콘텐츠화 방어 논리

#### 안전 영역 (Green Zone)

```
✅ F-Droid 오픈소스 코드를 화면에 띄우고 분석/설명
   → 라이선스가 명시적으로 허용

✅ "이 오픈소스 앱의 코드를 참고해서 내 앱을 이렇게 만들었다"
   → 교육적 콘텐츠 + 원저작자 크레딧

✅ 오픈소스 앱의 구조 다이어그램, 아키텍처 설명
   → 아이디어/구조는 저작권 대상이 아님

✅ 내 Parksy 앱 코드를 보여주면서 "이건 Ringdroid에서 영감 받았다"
   → 내 코드 + 출처 표시

✅ 디컴파일 과정 자체를 교육 목적으로 시연
   → 도구 사용법 교육은 합법
```

#### 위험 영역 (Red Zone)

```
❌ 상용 앱 디컴파일 코드를 화면에 통째로 노출
   → 저작물 공중 송신

❌ "이 앱의 핵심 알고리즘이 이거다" + 상세 코드 공개
   → 영업비밀 침해 가능

❌ "이렇게 하면 유료 기능 우회된다"
   → 기술적 보호 조치 무력화 교사

❌ 상용 앱 코드를 복사해서 내 앱에 넣고 배포
   → 저작권 침해
```

#### 최적 콘텐츠 전략

```
시리즈: "오픈소스로 나만의 Android 앱 만들기"

에피소드 구성:
1. "F-Droid에서 쓸만한 앱 5개 찾았다"
   → 앱 소개 + 설치 + 소스코드 링크

2. "소스코드 까보니 이렇게 만들었다"
   → 오픈소스 코드 화면 노출 OK + 분석 설명

3. "이걸 참고해서 내 앱에 적용해봤다"
   → Parksy 개발 과정 (내 코드 중심)

4. "삼성 앱은 이런 패턴을 쓰더라"
   → 개념만 말로 설명 (디컴파일 코드 직접 노출 X)

5. "결과물: Parksy Wavesy v2.0"
   → 완성된 앱 시연 + 비포/애프터
```

## 4. 개발 철학: "편집이 창작이다"

### 4.1 선언

> **표절부터 시작한다. 편집이 창작이다.**
> 베이스가 있으면 시간이 단축된다. 제로에서 시작하는 건 낭만이지 효율이 아니다.

이것은 소프트웨어 업계의 표준 관행이다. 이름만 다를 뿐이다:

| 업계 용어 | 의미 | 예시 |
|-----------|------|------|
| Fork | 남의 코드를 복사해서 내 것으로 발전 | Linux → Android |
| Leverage | 기존 솔루션 위에 구축 | WebKit → Chrome (Blink) |
| 바퀴를 재발명하지 마라 | 있는 걸 또 만들지 마라 | npm, pip 생태계 전체 |
| Standing on shoulders of giants | 거인의 어깨 위에 서라 | 뉴턴 이래 모든 과학 |

### 4.2 AI(Claude)의 역할 재정의

Claude Code가 0에서 1을 만드는 것보다, 기존 검증된 코드를 편집하는 게 품질이 높다.

| 작업 | AI가 0에서 짤 때 | 오픈소스 베이스 + AI 편집 |
|------|-----------------|----------------------|
| 코드 품질 | 컴파일은 됨, 실전 미검증 | 수천 명이 실기기에서 검증 |
| 엣지 케이스 | 상상으로 처리 | 실제 버그 리포트로 해결됨 |
| 기기 호환성 | 일반론적 처리 | 삼성/픽셀/샤오미 별 분기 존재 |
| 개발 속도 | 빠르나 디버깅이 길다 | 베이스가 안정적이라 디버깅 짧다 |
| 최적 용도 | 보일러플레이트, 글루코드 | **핵심 로직은 검증된 코드 활용** |

**Claude Code 최적 역할:**
1. 기존 코드 **읽고 이해**하기
2. 두 코드를 **합치기** (merge)
3. 특정 부분을 **교체/수정**하기
4. Parksy 브랜드로 **리스킨**하기
5. 보일러플레이트 **자동 생성**하기
6. 빌드/배포 **파이프라인 구성**하기

### 4.3 품질 등식

```
최종 품질 = 베이스 코드 품질 × 편집 정확도 × 테스트 깊이

- 베이스 = F-Droid 검증된 코드 (높음)
- 편집 = Claude Code AI (높음)
- 테스트 = 실기기 Termux 빌드 (실전)

>>> 0에서 AI가 짠 코드 (중간) × 1 × 테스트 (실전)
```

## 5. 도구 체인 (Toolchain)

### 5.1 소스 확보 도구

| 도구 | 용도 | Tier |
|------|------|------|
| F-Droid 앱 | 오픈소스 앱 발견 + 설치 | 1 |
| git clone | 소스코드 확보 | 1 |
| Samsung Developer Portal | 공식 SDK/문서 | 2 |
| ADB (android-tools) | APK 추출 | 3 |
| jadx / jadx-gui | APK → Java 소스 디컴파일 | 3 |

### 5.2 APK 추출 파이프라인 (Tier 3 한정)

```bash
# Termux 환경

# 1. ADB 연결 (무선 디버깅)
adb pair 192.168.x.x:PORT    # 페어링 (최초 1회)
adb connect 192.168.x.x:PORT # 연결

# 2. 패키지 검색
adb shell pm list packages | grep -i "키워드"

# 3. APK 경로 확인
adb shell pm path com.example.app

# 4. APK 추출
adb pull /data/app/.../base.apk ~/apks/appname.apk

# 5. 디컴파일
jadx -d ~/decompiled/appname ~/apks/appname.apk

# 6. 분석
cat ~/decompiled/appname/resources/AndroidManifest.xml  # 구조 파악
grep -r "키워드" ~/decompiled/appname/sources/          # 코드 검색
```

### 5.3 원스크립트 자동화

```bash
#!/data/data/com.termux/files/usr/bin/bash
# extract-apk.sh <package_name>
PKG=$1
NAME=$(echo $PKG | rev | cut -d. -f1 | rev)
mkdir -p ~/apks/$NAME

echo "=== $PKG 추출 중... ==="
adb shell pm path $PKG | sed 's/package://' | while read p; do
  adb pull "$p" ~/apks/$NAME/
done

echo "=== jadx 디컴파일 중... ==="
jadx -d ~/decompiled/$NAME ~/apks/$NAME/base.apk

echo "=== 완료: ~/decompiled/$NAME/ ==="
```

## 6. 라이선스 컴플라이언스 체크리스트

Parksy 앱 배포 전 반드시 확인:

```
□ 사용한 오픈소스 라이브러리 목록 정리
□ 각 라이브러리의 라이선스 유형 확인
□ Apache 2.0: NOTICE 파일 포함 여부
□ MIT/BSD: 저작권 고지 포함 여부
□ GPL 코드: 직접 포함하지 않았는지 확인 (참조만 했는지)
□ Tier 3 코드: 복사하지 않았는지 확인 (개념 참조만 했는지)
□ 앱 내 "오픈소스 라이선스" 화면 구현 여부
□ CREDITS.md 또는 THIRD_PARTY_LICENSES.md 작성 여부
```

## 7. 위험 관리 매트릭스

| 리스크 | 확률 | 영향 | 대응 |
|--------|------|------|------|
| GPL 코드 실수로 포함 | 중 | 높음 | 소스 공개 또는 해당 코드 제거 |
| 상용 앱 코드 복사 발각 | 낮음 | 높음 | Tier 3은 개념 참조만, 코드 복사 금지 원칙 |
| 라이선스 고지 누락 | 중 | 중 | 배포 전 체크리스트 의무화 |
| YouTube에서 저작권 클레임 | 낮음 | 중 | 오픈소스 코드만 화면 노출, 상용 앱은 개념만 설명 |
| 특허 침해 | 낮음 | 높음 | 개인 비상업 배포로 리스크 최소화 |

## 8. 결론: 3줄 요약

1. **소스풀 = F-Droid (Primary) + 삼성 공식 (Secondary) + 디컴파일 참조 (Tertiary)**
2. **프로세스 = 오픈소스 clone → 라이선스 확인 → AI 편집 → Parksy 통합 → 실기기 빌드**
3. **법적 방어 = 오픈소스 라이선스 준수 + 디컴파일은 개념 참조만 + YouTube는 오픈소스 코드만 노출**

> **"바퀴를 재발명하지 마라. 좋은 바퀴를 가져와서, 내 차에 맞게 깎아라."**
> **그것이 포크이고, 그것이 편집이고, 그것이 창작이다.**
